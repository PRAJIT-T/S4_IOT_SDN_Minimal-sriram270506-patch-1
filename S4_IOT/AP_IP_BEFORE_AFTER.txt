HOW IP DETECTION WORKS
══════════════════════

get_actual_ip() Method:
────────────────────────

def get_actual_ip(self):
    """Get the actual IP address of this machine"""
    try:
        # Connect to controller to determine our IP
        s = socket.socket(socket.AF_INET, socket.SOCK_DGRAM)
        s.connect((self.controller_ip, self.controller_port))
        ip = s.getsockname()[0]           ← ✓ KEY: OS determines IP
        s.close()
        return ip
    except:
        try:
            return socket.gethostbyname(socket.gethostname())
        except:
            return "127.0.0.1"

Why this works:
  1. Create UDP socket
  2. "Connect" to controller (UDP doesn't handshake, but routes)
  3. Ask OS: "Which IP would I use for this connection?"
  4. getsockname()[0] returns exactly that
  5. This is ACTUAL IP kernel determined


DEMO: How IP Detection Works
═════════════════════════════

Single Laptop (127.0.0.1):
──────────────────────────

$ python3 ap_agent_multi_laptop.py --controller 127.0.0.1

Step 1: AP creates UDP socket
Step 2: AP connects socket to 127.0.0.1:9000
Step 3: OS says: "To reach 127.0.0.1, you use 127.0.0.1"
Step 4: getsockname() returns 127.0.0.1
Step 5: Output: [AP Agent] Actual IP Address: 127.0.0.1 ✓


Two Laptops (Different Subnets):
─────────────────────────────────

Laptop B (192.168.1.101):
$ python3 ap_agent_multi_laptop.py --controller 192.168.1.100

Step 1: AP creates UDP socket
Step 2: AP connects socket to 192.168.1.100:9000
Step 3: OS says: "To reach 192.168.1.100, you use interface 192.168.1.101"
Step 4: getsockname() returns 192.168.1.101
Step 5: Output: [AP Agent] Actual IP Address: 192.168.1.101 ✓

This is the ACTUAL IP that controller will see!


Why This Is Better Than Alternatives
──────────────────────────────────────

Method 1: gethostbyname(gethostname()) ❌
  Problem: Doesn't consider routing
  Result: Might return loopback (127.0.0.1) even on multi-laptop
  
Method 2: Parse ifconfig output ❌
  Problem: Complex, OS-dependent
  Result: Picks first interface, might be wrong
  
Method 3: Our getsockname() approach ✓
  Solution: OS determines actual routing
  Result: Always correct for the controller we're connecting to
