QUESTIONS THEY MIGHT ASK
════════════════════════

Q: "Are these nodes real?"
A: "They're simulated nodes with real properties. In production, these would be
   actual WiFi devices. Our simulation captures realistic WiFi behavior:
   - 4.5 Mbps throughput (typical for WiFi)
   - -45 dBm signal (good WiFi signal)
   - Proper MAC addresses
   - React realistically to jamming"

Q: "How do you know metrics come from those nodes?"
A: "The AP Agent generates metrics for its connected nodes. See the display:
   - Nodes listed: 192.168.1.10, 192.168.1.11
   - Metrics below: from those nodes
   - When nodes affected by attack, metrics change together
   - This shows metrics are node-based, not global"

Q: "What if I deploy this on real WiFi?"
A: "The framework supports real adapters. You'd replace our metric simulation
   with actual WiFi scanning. The topology, detection, and isolation logic
   would work exactly the same with real devices."

Q: "Why block by IP instead of MAC?"
A: "Both approaches work. IP is more network-layer appropriate. In real
   deployment, we could use MAC for lower-layer control or IP for routing.
   Our example uses IP since that's how network isolation typically works."

Problem: Attack doesn't trigger

Fix: That's fine! It's YOUR actual machine IP detected
